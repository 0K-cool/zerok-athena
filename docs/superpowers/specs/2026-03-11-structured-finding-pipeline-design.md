# Structured Finding Pipeline — Design Spec

**Date:** 2026-03-11
**Status:** Approved
**Author:** Vex (brainstorming session with Kelvin)
**Research:** `output/research/2026-03-11-multiagent-pentest-finding-pipeline-research.md`

## Problem

ATHENA's current finding extraction pipeline (`extract_findings()` in `message_bus.py`) uses regex keyword matching against ALL tool output. This produces false positives when agents read playbooks, query Neo4j, call internal APIs, or read documentation. Every false positive erodes trust in the dashboard.

The current mitigation is a growing blacklist of tool names and content patterns to skip — whack-a-mole that doesn't scale.

## Solution

Replace the regex-on-everything approach with a **structured-first pipeline (A+)**:

1. **Agent self-reporting** is the primary finding source (structured, intentional, contextual)
2. **Scanner extractor** is the safety net — only runs on output from known scanner commands (whitelist)
3. **Catch-all heuristic** handles novel tools/zero-days — triggers only on unambiguous strong signals
4. **Confidence levels** gate what gets broadcast vs logged vs ignored

## Architecture

```
Tool Output (Bash, MCP, Kali execute)
  │
  ├─ Known scanner command? (whitelist check)
  │     YES → Scanner Extractor
  │            → Structured Finding with confidence HIGH or MEDIUM
  │     NO  → Strong signal check (catch-all heuristic)
  │              YES → Finding with confidence LOW (dashboard only)
  │              NO  → Skip entirely
  │
  ├─ Agent self-reports via POST /api/bus/publish
  │     → Schema validation → confidence HIGH → Bus
  │
  └─ Bus receives Finding
        → Dedup check (hash of finding_type + target + summary[:50])
        → Confidence gate:
              HIGH   → Broadcast to agents + Neo4j + Dashboard (full color)
              MEDIUM → Broadcast to agents + Neo4j + Dashboard (full color)
              LOW    → Dashboard only (dimmed, "unvalidated" badge)
```

## Design Decisions

### Why whitelist instead of blacklist

The current blacklist (`_SKIP_TOOLS`) requires a new entry for every non-scanner tool that produces false positives. The whitelist approach inverts this: only ~25 known scanner commands are extracted. Everything else is ignored by default. New tools installed on Kali work automatically via agent self-reporting — no config change needed.

### Why agent self-reporting is primary

Agents are Claude instances — the best LLM available. They understand context that regex never will: a playbook describing SQL injection vs an actual SQL injection found in a target. For novel attacks and zero-days, only the agent can recognize the finding. The extractor cannot discover what it has no pattern for.

### Why three confidence levels (not decimal scores)

Decimal confidence (0.85 vs 0.80) implies false precision. The system needs three decisions: broadcast it, show it, or ignore it. Three levels map directly to three actions. Add granularity later when real usage patterns demand it.

### Why two states (not four)

`discovered` and `exploited` cover current engagement patterns (minutes, not days). `confirmed` and `reported` are lifecycle states for a mature product with finding management. YAGNI — add when needed.

## Finding Schema

```python
{
    # Identity
    "finding_type": "open_port|cve|vulnerability|credential|shell|service|network",
    "dedup_key": "<auto-generated hash if omitted>",

    # Trust
    "confidence": "high|medium|low",
    "source": "agent|extractor",
    "state": "discovered|exploited",

    # What
    "summary": "OpenSSH 8.9p1 on port 22",
    "severity": "critical|high|medium|low",
    "target": "10.0.0.1",
    "port": 22,                          # Optional
    "service": "ssh",                    # Optional
    "cve": ["CVE-2023-12345"],           # Optional
    "cwe": "CWE-89",                     # Optional
    "technique": "T1190",                # MITRE ATT&CK, optional

    # Proof
    "evidence": {
        "tool": "nmap",
        "command": "nmap -sV 10.0.0.1",
        "output": "22/tcp open ssh OpenSSH 8.9p1"
    },

    # Action
    "action_needed": "Check for CVE-2023-12345"
}
```

The `BusMessage` dataclass remains the transport wrapper. The Finding schema lives in the `data` field, with `bus_type="finding"`.

## Confidence Assignment

| Source | Signal | Confidence |
|--------|--------|-----------|
| Agent self-report (any) | Agent understood and published | HIGH |
| Scanner extractor | CVE match | HIGH |
| Scanner extractor | Open ports | MEDIUM |
| Scanner extractor | Service/version | MEDIUM |
| Scanner extractor | Keyword vuln match | MEDIUM |
| Catch-all heuristic | Root shell, meterpreter, SYSTEM | LOW |
| Catch-all heuristic | CVE pattern in unknown command | LOW |
| Catch-all heuristic | Credential indicator | LOW |

## Confidence Gates

| Level | Broadcast to agents | Neo4j persist | Dashboard | Notes |
|-------|-------------------|---------------|-----------|-------|
| HIGH | Yes | Yes | Full color | Trusted finding |
| MEDIUM | Yes | Yes | Full color | Scanner-confirmed |
| LOW | No | Yes (flagged) | Dimmed + "unvalidated" badge | Safety net only |

## Scanner Whitelist

```python
SCANNER_COMMANDS = {
    # Recon
    "nmap", "naabu", "masscan", "subfinder", "amass", "httpx",
    # Web scanning
    "nuclei", "nikto", "gobuster", "feroxbuster", "ffuf", "wpscan", "dirb",
    # Exploitation
    "sqlmap", "hydra", "crackmapexec", "metasploit", "msfconsole",
    # Network
    "enum4linux", "smbclient", "rpcclient", "snmpwalk",
    # SSL/TLS
    "testssl", "sslscan",
}
```

Detection: extract the first token from the Bash command string. For piped commands (`nmap ... | grep`), check the first command. For Kali API calls, extract the tool name from the command payload.

Extensible: add new tools with one line. But not required — unlisted tools work via agent self-reporting.

## Catch-All Strong Signals

```python
STRONG_SIGNALS = {
    r"uid=\d+\(root\)":       ("shell", "low"),
    r"CVE-\d{4}-\d{4,7}":     ("cve", "low"),
    r"root@[\w.-]+[:#]":      ("shell", "low"),
    r"NT AUTHORITY\\SYSTEM":   ("shell", "low"),
    r"meterpreter\s*>":       ("shell", "low"),
}
```

Five patterns. Unambiguous in any context — a root shell is a root shell regardless of what tool produced it. Never false positive on documentation or playbook content.

## Dedup Logic

When a finding arrives at the bus:

1. Compute `dedup_key`: agent-provided, or `hash(finding_type + target + summary[:50])`
2. Check in-memory dict (keyed by `dedup_key`, scoped to engagement)
3. If new → store and process
4. If duplicate → merge: keep higher confidence, promote state forward (discovered → exploited), keep most recent evidence. Re-broadcast only if state changed.

## Zero-Day / Novel Tool Coverage

The pipeline handles unknown tools through two mechanisms:

1. **Agent self-reporting** — Claude can reason about any tool output, including tools that don't exist yet. If an agent installs a new tool on Kali and finds a vulnerability, it self-reports via the bus API. No configuration change needed.

2. **Catch-all strong signals** — Five unambiguous patterns (root shell, meterpreter, SYSTEM, CVE pattern) fire regardless of source tool. These never false-positive on documentation.

The extractor was never going to discover a zero-day — there's no regex for something nobody's seen before. That's an LLM reasoning task, and the agents are the LLMs.

## Success Criteria

1. Zero false positives from playbook reads, Neo4j queries, API responses, and documentation content
2. All scanner findings (ports, CVEs, services) still detected automatically
3. Agent self-reported findings arrive on dashboard within 2 seconds
4. Novel tool findings work via agent self-reporting without config changes
5. Dashboard distinguishes validated vs unvalidated findings visually
6. Neo4j attack graph builds correctly from structured Finding objects
