# BUG: VF PATCH /findings/{fid} Returns 500 — No Endpoint for Finding Updates

**Date:** March 27, 2026
**Severity:** HIGH — VF can't confirm exploits through the proper channel
**Status:** DOCUMENTED — Not yet fixed

## Problem

VF agent tries to PATCH existing findings to set `status: "confirmed"` but gets HTTP 500 Internal Server Error. VF is doing the right thing — attempting to update existing findings instead of creating duplicates (MAPTA PoC-gate model) — but the server doesn't support PATCH on findings.

## Evidence

```
VF THOUGHT: "The PATCH didn't seem to update the status. Let me check what the PATCH endpoint actually requires."

VF → curl -s "http://localhost:8080/openapi.json" | python3 -c "..."
OUTPUT: {} (empty — no PATCH schema)

VF → curl -s -X PATCH "http://localhost:8080/api/engagements/eng-eb2adc/findings/a14b4f62" \
     -H "Content-Type: application/json" \
     -d '{"status":"confirmed"}'
OUTPUT: < HTTP/1.1 500 Internal Server Error
```

## Impact

- VF can't update finding status to "confirmed" via PATCH
- VF falls back to creating new findings (duplicates)
- The dedup pipeline catches some via fingerprint MERGE, but not all
- Confirmed Exploit Rate may be inaccurate

## Proposed Fix

Two options:

### Option A: Add PATCH endpoint (recommended)
```python
@app.patch("/api/engagements/{eid}/findings/{fid}")
async def update_finding(eid: str, fid: str, request: Request):
    payload = await request.json()
    # Update status, verified, evidence, confirmed_at on existing Finding node
    # Set confirmed_at when status transitions to "confirmed"
```

### Option B: Instruct VF to use POST /api/findings
VF prompt change: "To confirm an exploit, POST to /api/findings with the same title and target — the MERGE pipeline will update the existing finding and set confirmed_at."

### Recommendation
Option A is cleaner — PATCH is the semantically correct HTTP verb for updating a resource. VF already knows the finding ID, so it should be able to update it directly.
